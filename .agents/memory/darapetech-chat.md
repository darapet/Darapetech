# Darapetech Chat System

## Key Architecture

**Firestore collections:**
- `users` — user accounts (uid, name, email, initials)
- `agents` — agent profiles (id=docId, agentId=DT-001, uid, name, email, service, skills, photoUrl, bio, active, firstLogin)
- `agent_conversations` — one doc per agent-user pair (agentId, userId, lastMessage, unreadAgent, unreadUser)
- `agent_messages` — all messages (conversationId, senderId, senderType, type, content/fileUrl/fileName)

**Auth separation:**
- Users: Firebase email/password auth
- Agents: same Firebase auth but identified by having an `agents` collection doc with matching uid
- Admin: Google auth (email `daramolapeter98@gmail.com`)
- Agent creation uses a secondary Firebase app (`firebase.initializeApp(config, 'agentCreator')`) to avoid logging out the admin

**Agent IDs:** sequential `DT-001` format, generated by reading the last agent doc ordered by createdAt

**File uploads:** Firebase Storage at `chat/{convId}/...` for messages, Cloudinary for agent profile photos

**firstLogin flow:** Agent created with `firstLogin: true`; agent-login.html forces password change modal on first login, sets `firstLogin: false` after

**Photo storage:** Agent photos → Cloudinary (uses settings/cloudinary doc); Chat files → Firebase Storage

**Why secondary app for agent creation:** `auth.createUserWithEmailAndPassword()` logs in the new user, which would sign out the admin. The secondary app isolates this side effect.

**How to apply:** Any future code that creates Firebase Auth users on behalf of another user must use the `getAgentCreatorApp()` pattern in firebase-app.js.
